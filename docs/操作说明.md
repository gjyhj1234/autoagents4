# 全流程操作说明（零基础向）

> 本文档配套仓库里的 `src/GitHubAutoApprove`（WinForm 工具）和 `.github/workflows/*`（演示 workflow）。  
> 目标：**让 Copilot 编码代理自动开 PR → CI 自动跑 → 卡在审批时 WinForm 自动点 Approve → 测试通过自动合并 → 关闭 Issue → 触发下一个子任务，全程无人干预。**

本文档所有的截图文字、按钮文字来自 GitHub 官方界面（2025 年版）。如果你看到的文字有略微差异，基本都是同义词（例如 `Review pending deployments` 和 `Review deployments` 会混用），不影响操作。

---

## 0. 整体架构

```
  ┌────────────────┐     1. 创建 Issue (指派给 Copilot)
  │  你 (浏览器)   │─────────────────────┐
  └────────────────┘                     ▼
                               ┌────────────────────┐
                               │  Copilot 编码代理  │
                               │  (GitHub 托管)     │
                               └─────────┬──────────┘
                        2. 新分支+PR     │
                                         ▼
                               ┌────────────────────┐
                               │  GitHub Actions    │
                               │ copilot-task-demo  │
                               └─────────┬──────────┘
                     3. 跑到 environment │  ← 这里卡住，等审批
                        "production-     │
                        approval" 暂停   ▼
  ┌────────────────┐  4. 轮询发现 waiting 的 run
  │ WinForm        │◀────────────────────┤
  │ (WebView2)     │─── 5. 模拟点击 ────▶│
  │ 登录为仓库管理 │    Approve and run  │
  │  员账号        │                     ▼
  └────────────────┘              Workflow 继续运行
                                         │
                               6. 全部通过 ▼
                               ┌────────────────────┐
                               │ auto-merge-and-    │
                               │   chain workflow   │
                               └─────────┬──────────┘
                  7. squash merge PR     │
                  8. 关闭 Issue          │
                  9. 给"下一个子任务"    │
                     打 status:ready     ▼
                                 ┌────────────────┐
                                 │ 下一个子任务  │
                                 │ 重新循环      │
                                 └────────────────┘
```

## 1. 前置条件

| # | 条件 | 如何检查 |
|---|------|---------|
| 1 | 你有一个 GitHub 账号，并且能访问 github.com | 打开 https://github.com 能登录即可 |
| 2 | 已订阅 **GitHub Copilot**（Individual / Business / Enterprise 都可以） | `Settings → Billing and plans → Plans and usage → Copilot` 能看到「Active」 |
| 3 | 已启用 **Copilot coding agent**（编码代理） | 账号页 `Settings → Copilot → Coding agent`，把开关打开。企业 / 组织版由 org owner 在 `Organization settings → Copilot → Policies` 里开启 |
| 4 | Windows 10/11，有 .NET 8 SDK 和 WebView2 Runtime | WinForm 工具需要 |

> **关于「安全不是当前考虑范畴」**：本方案会让一个有仓库写权限的账号**自动批准**所有审批请求。生产环境里这会绕过审批审查，请务必只在测试仓库使用。

## 2. 创建并初始化仓库

### 2.1 新建仓库

1. 右上角 `+` → `New repository`
2. Owner 选你自己的用户名，Repository name 随意（例如 `copilot-autoflow-demo`）
3. Visibility：**Private 或 Public 都行**（Copilot 编码代理两者都支持）
4. 勾选 `Add a README file`，`Create repository`

### 2.2 把本仓库的演示代码复制进去

最简单：直接把本仓库整个 clone 过去，或者只拷贝下面三个目录到你的新仓库：

```
.github/workflows/copilot-task-demo.yml
.github/workflows/auto-merge-and-chain.yml
.github/ISSUE_TEMPLATE/copilot-subtask.yml
```

也可以直接 **Fork 本仓库** 进行试验。

## 3. 配置仓库权限（关键 ⚠️）

这一节解决「为什么每次都要人工点 Approve」的根本问题。请 **严格按顺序** 配置。

打开你的仓库 → `Settings`（顶部导航栏最右边）。

### 3.1 Actions 的默认权限

进入 `Settings → Actions → General`，按下图配置：

| 项目 | 推荐值 | 说明 |
|------|--------|------|
| **Actions permissions** | `Allow all actions and reusable workflows` | 否则用不了 `actions/github-script` |
| **Fork pull request workflows from outside collaborators** | `Require approval for first-time contributors` | 如果只是你自己一个人玩，保持默认即可。想彻底关掉人工审批请选 `Require approval for first-time contributors who are new to GitHub`（最宽松） |
| **Workflow permissions** | `Read and write permissions` | 自动合并 / 关闭 Issue 必须写权限 |
| ✅ **Allow GitHub Actions to create and approve pull requests** | **勾上** | 允许 Actions 把 PR 批准或合并 |

最下面点 `Save`。

### 3.2 允许 Copilot coding agent 触发 Actions

进入 `Settings → Code & automation → Actions → General` 页面底部，找到
**"Allow GitHub Actions to approve pull requests"**——确保勾选。

另一个坑：**默认情况下，Copilot agent 开的 PR 触发的 workflow 第一次也会卡在 "Approve and run"**（因为 GitHub 把 `copilot-swe-agent[bot]` 视为"外部贡献者"）。解决方法就是 3.1 里的那个下拉框选成最宽松选项，或者用 WinForm 工具每次自动点一下。

### 3.3 创建带审批的 Environment（**这就是需要人工点击的来源**）

进入 `Settings → Environments → New environment`。

1. Name 填 `production-approval`（必须与 `copilot-task-demo.yml` 里的 `environment:` 一致）
2. 点 `Configure environment`
3. 勾选 **Required reviewers**，把**你自己**添加为审批者
4. （可选）勾选 `Prevent self-review` → **不要勾**，因为我们就要自审
5. `Save protection rules`

> 现在，任何 workflow 进入这个 environment 时都会暂停，并在 Actions 页面显示 **Review deployments** 或 **Approve and run** 按钮。我们的 WinForm 工具就是专门去点它。

### 3.4 创建两个标签

进入 `Issues → Labels → New label`，分别创建：

- `status:ready`（绿色）— 表示一个子任务可以开始
- `copilot-task`（蓝色）— 标识这是交给 Copilot 的任务

## 4. 把 Copilot 编码代理接入仓库

1. 打开 `https://github.com/settings/copilot/coding-agent`，确认总开关打开。
2. 在仓库里：右上角头像 → `Your repositories` → 选择本仓库 → `Settings → Copilot → Coding agent` → 勾选 **Enable coding agent for this repository**（组织仓库需要先由 Org 放行）。
3. 验证：进入 Issues 页面，在任意 Issue 右侧 Assignees 下拉框里应该能搜到 **Copilot**（带有彩色 logo 的虚拟用户）。

## 5. 准备 WinForm 工具

（详细说明见 [`src/GitHubAutoApprove/README.md`](../src/GitHubAutoApprove/README.md)）

### 5.1 编译

```cmd
cd src\GitHubAutoApprove
dotnet build -c Release
```

或 Visual Studio 2022 打开 `.sln` 直接 F5。

### 5.2 首次登录

1. 启动 `GitHubAutoApprove.exe`。
2. 程序顶部是内嵌 Edge 浏览器，会自动打开 `github.com/login`。
3. 用**仓库管理员账号**登录（必须有写权限，否则根本看不到 Approve 按钮）。
4. 登录后 cookie 会持久化在 `%AppData%\GitHubAutoApprove\WebView2`，下次打开就不用再登录。

### 5.3 配置仓库并开始轮询

1. 顶部工具栏：
   - `Owner` 填你的用户名/组织名
   - `Repo` 填仓库名
   - `间隔(秒)` 默认 30（建议保持）
2. 点 **保存配置**。
3. 点 **▶ 开始轮询**。

工具会每 30 秒执行一次：

1. 导航到 `https://github.com/<owner>/<repo>/actions?query=is:waiting`
2. 用 JS 抓到所有卡在等待的 run
3. 依次打开每个 run 的详情页，点击 `Approve and run` / `Approve and deploy` 按钮
4. 所有动作写进下半部的日志框

## 6. 端到端演示

现在真正跑一遍。

### 6.1 创建演示 Issue

1. 进入 `Issues → New issue`，选模板 **🤖 Copilot 子任务**。
2. 标题：`[sub-task] 生成 docs/hello.md`
3. 目标一句话：`在 docs/ 目录创建 hello.md`
4. 详细要求：
   ```
   1. 新建 docs/hello.md
   2. 文件内容为单行 "# Hello from Copilot"
   3. 确保这是一个 git 新文件，不修改其它任何文件
   ```
5. 下一个子任务：留空（或指向另一个准备好的 Issue，例如 `#2`）
6. 右侧 `Assignees` 选 **Copilot**
7. 提交

### 6.2 观察 Copilot agent 工作

- 几秒到几分钟后，Issue 下方会出现 Copilot 的 workflow run（"Copilot is on the job"）。
- 它会：
  1. 创建新分支 `copilot/<hash>`
  2. 在分支上提交 `docs/hello.md`
  3. 发起一个 Draft PR，标题类似 `[WIP] 生成 docs/hello.md`，body 包含 `Closes #<issue_number>`
- 进入 PR 页面，你会看到：
  - ✅ `copilot-task-demo / build-and-test` 已经跑完
  - ⏸ `copilot-task-demo / require-approval` **Waiting for review**（需要审批 production-approval 环境）

### 6.3 WinForm 自动点击

- 切回 WinForm 工具，观察日志：
  ```
  [10:03:12] [扫描] 打开 https://github.com/<you>/<repo>/actions?query=is:waiting
  [10:03:15] [扫描] 发现 1 个可能等待审批的 run: 123456
  [10:03:15] [审批] 打开 https://github.com/<you>/<repo>/actions/runs/123456
  [10:03:18]   ✔ run #123456 点击了 [Review deployments] 按钮。等待确认对话框...
  [10:03:20]   确认阶段: confirmed
  ```
- 回到浏览器刷新 PR：`require-approval` 会变成 ✅ 通过。

### 6.4 自动合并 + 自动关闭 Issue + 自动触发下一个任务

- `copilot-task-demo` 全部成功后，`auto-merge-and-chain` workflow 会被 `workflow_run` 事件触发：
  1. `github.rest.pulls.merge` 用 squash 合并 PR
  2. 从 PR body 里匹配 `Closes #N`，调用 `issues.update` 关闭对应 Issue
  3. 读取该 Issue body，如果写了「下一个子任务 `#M`」，就给 Issue `#M` 贴上 `status:ready` 标签
- Copilot coding agent 可配置为看到 `status:ready` 标签就接手（见 [GitHub 官方文档](https://docs.github.com/en/copilot/how-tos/agents/coding-agent)），或者你也可以在 `auto-merge-and-chain.yml` 里追加一步调用 `issues.addAssignees` 把 Copilot 加回去。

## 7. 常见问题

### Q1：WinForm 里看不到 Approve 按钮？
- 你登录的账号必须在 environment 的 Required reviewers 列表里。
- 把仓库 Owner/Repo 填错也会显示不到（URL 404）。

### Q2：`Approve and run` 按钮**从来不出现**，PR 上显示绿色 "All checks have passed"？
- 说明 workflow 根本没有卡点。常见原因：
  1. Environment 名字拼错（workflow 里是 `production-approval`，但你建的叫 `production_approval`）
  2. 没有为环境配置 Required reviewers

### Q3：WinForm 日志显示 "检测到当前未登录"？
- WebView2 cookie 过期。点工具栏「打开 GitHub 登录」重新登录一次即可。

### Q4：`auto-merge-and-chain` 没触发？
- `workflow_run` 事件只有在 **默认分支** 上存在该 workflow 文件时才会触发。确保 `auto-merge-and-chain.yml` 已经合并到 `main`（第一次可以直接提到 main 上，不要走 PR）。
- 查看 `Actions → auto-merge-and-chain → 最近的 run`，如果有日志就能看到失败原因。

### Q5：合并被 branch protection 拦下？
- `Settings → Branches → Branch protection rules`，要么允许 Actions 绕过规则（勾 `Allow specified actors to bypass required pull requests`，把 `github-actions[bot]` 加进去），要么先降低保护再做演示。

### Q6：我想**彻底取消审批**，不用 WinForm？
- 把 `copilot-task-demo.yml` 里 `environment: production-approval` 这一行删除即可。但那样就没法演示 WinForm 自动审批了——WinForm 的价值正是在**确实需要留审批**但你不想每次手点的场景。

### Q7：有没有更"正统"的方案，不用模拟点击？
- 有——用 Personal Access Token (classic，勾 `repo` + `workflow`) 调用 REST API：
  - 批准 environment：`POST /repos/{owner}/{repo}/actions/runs/{run_id}/pending_deployments`
  - 批准 first-time contributor：`POST /repos/{owner}/{repo}/actions/runs/{run_id}/approve`
- 但题目明确要求**用 WebView2 模拟人工点击**，所以主方案就是现在这个 WinForm 工具。REST API 方案可作为备选（后续可以在工具里加一个"API 模式"开关，留作扩展）。

## 8. 安全性声明（再次强调）

- 该 WinForm 工具会自动把**所有**等待审批的 run 批准。
- 它仅在本地运行，不会把 cookie / token 发送到任何第三方。
- 千万不要在"真实生产" environment 上用，它绕过了审批的初衷。
- 如果误审，可以在 Actions 页面的 run 详情里随时 `Cancel workflow`。
